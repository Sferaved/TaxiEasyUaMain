## 2. Старый способ (карта WFP по токену) vs Google Pay

### 2.1. Сравнение для пользователя

| | **Карта WFP (`wfp_payment`)** | **Google Pay (`google_pay_payment`)** |
|---|-------------------------------|----------------------------------------|
| Подготовка | Нужна привязка карты в WebView WayForPay (`rectoken` в приложении) | Привязка **не нужна** — карта берётся из **Google Wallet** |
| Момент оплаты | Заказ уходит **сразу**; холд на сервере **после** создания заказа | Сначала **шторка Google Pay** и холд; заказ — **только после успеха** |
| Что подтверждает пользователь | (При первой привязке — WebView банка; при заказе — без отдельного шага, если карта уже есть) | Каждый заказ: **отпечаток / PIN / «Оплатить»** в шторке GPay |
| Сумма в момент оплаты | Холд на полную стоимость заказа | То же — холд на полную стоимость заказа |

### 2.2. Сравнение техническое (для тестировщика)

| Шаг | **Карта по токену** | **Google Pay** |
|-----|---------------------|----------------|
| 1 | Генерируется `orderReference` → в URL как `wfpInvoice` | То же |
| 2 | `orderClientCostMyApi` → заказ в БД и диспетчер | **Пока не вызывается** |
| 3 | Сервер: `orderIdMemoryToken` + **`chargeActiveToken(rectoken)`** | Приложение: `GET googlePayConfig` → шторка GPay → **`POST googlePayCharge(gpToken)`** |
| 4 | WFP: холд по сохранённому токену карты | WFP: холд по токену Google Pay |
| 5 | Push / poll статуса (`SimplePollStatusJob` и т.д.) | После успеха холда → **тот же** `orderClientCostMyApi` с тем же `wfpInvoice` |
| 6 | Сервер: `orderIdMemoryToken`, **без** второго charge для GPay | `orderIdMemoryToken`, poll статуса — **без** `chargeActiveToken` |

**Важно:** второй холд по **активной WFP-карте** при GPay **не выполняется** — иначе было бы двойное списание.

### 2.3. Заказ в диспетчере (вилка безнал + нал)

Поведение **как у карты по токену**:

- `authorizationChoiceApp('google_pay_payment')` → **`payment_type = 1`** (безнал).
- Если для города настроена **вилка** (`authorizationBonus` + `authorizationDouble`):
  - **безнальная нога** (поиск с оплатой картой/GPay);
  - **наличная нога** (дубль для алгоритма вилки).
- Это **не** два списания: один холд GPay + две заявки в диспетчере для поиска машины.

**Исключение:** если внешний диспетчер недоступен и заказ идёт только через **`my_server_api`** — создаётся **один** заказ без вилки (`doubleOrder = 0`), как и для карты в этом режиме.

### 2.4. TEST vs PRODUCTION (Google Pay API)

| Условие | Режим GPay | Что видит пользователь |
|---------|------------|-------------------------|
| Эмулятор | **TEST** | «Test Card: Visa •••• 1111», текст «тестовое средовище» |
| Телефон + **debug** APK | **TEST** | То же |
| Телефон + **release** APK (магазин / signed) | **PRODUCTION** | Реальная карта из Google Wallet, **без** предупреждения о тесте |

В logcat: `WfpGooglePayHelper: Google Pay environment: TEST` или `PRODUCTION`.

**Приёмка этапа 2 на реальном устройстве — только release 1.1030+, не debug из Android Studio.**

## 3. Требования перед тестом

### 3.1. Устройство

- Android с **Google Play Services** и **Google Wallet**.
- В Wallet добавлена **банковская карта** (Visa/Mastercard).
- Интернет стабильный.

### 3.2. Приложение

- **PAS_1 версия 1.1030** или новее, **release**-сборка.
- Не debug — иначе GPay останется в TEST.

### 3.3. Сервер

- Задеплоен коммит taxi_repo с поддержкой `google_pay_payment` (ветка `chargeActiveToken` только для `wfp_payment`, `authorizationChoiceApp` для GPay).
- WayForPay: для merchant города **включён Google Pay** (иначе после шторки возможен отказ WFP, напр. `reasonCode` 1129).

### 3.4. Настройки в приложении

- Город с **безналом** (как для WFP-карты), напр. OdessaTest для внутренних тестов.
- Способ оплаты: **Google Pay** (не «Картой» WFP, не нал).

## 4. Пошаговая проверка на реальном устройстве

### Сценарий A — успешный заказ

1. Открыть **заказ с карты** (Visicom).
2. Указать маршрут, дождаться стоимости.
3. Убедиться: способ оплаты — **Google Pay**.
4. Нажать **«Заказать»**.
5. **Ожидание:** открывается **нативная шторка Google Pay**:
   - сумма совпадает с заказом;
   - **ваша** карта из Wallet (не Test Card 1111);
   - **нет** красной надписи про тестовое средовище.
6. Подтвердить оплату (биометрия / PIN / «Оплатить»).
7. **Ожидание:**
   - переход к экрану поиска машины / активного заказа;
   - заказ в диспетчере как **безнал**;
   - при вилке — безнальная + наличная ноги (как у WFP-карты).
8. Заказ **не** должен создаваться, если шаг 5–6 не выполнен.

### Сценарий B — отмена GPay

1. Повторить шаги 1–4.
2. Закрыть шторку GPay или «Отмена».
3. **Ожидание:** остаётесь на экране заказа, toast об отмене/ошибке, **заказ не создан**.

### Сценарий C — нет двойного холда по WFP-карте

1. В приложении **привязана** WFP-карта (rectoken есть).
2. Способ оплаты — **Google Pay**, не карта.
3. Успешно оформить заказ (сценарий A).
4. **Ожидание:** один холд по GPay (`orderReference` из заказа); **нет** второго charge по rectoken активной карты.

### Сценарий D — что НЕ тестируем в 1.1030

| Не проверять | Почему |
|--------------|--------|
| Заказ с **HomeFragment** | GPay при заказе там не включён |
| **Доплата** в активном заказе через GPay | Этап 3, не реализован |
| Смена способа на GPay в шторке без нового заказа | Только запись в БД, без холда |

## 5. Что смотреть в логах (для разработки)

### Android (logcat)

Фильтр: `WfpGooglePayHelper`, `GooglePayOrderHelper`, `VisicomFragment`.

Успех:

```
Google Pay environment: PRODUCTION
GooglePayOrderHelper: googlePayCharge status=WaitingAuthComplete ref=V_...
VisicomFragment: orderFinished: ... orderClientCostMyApi ...
```

Отмена / ошибка:

```
Google Pay hold failed: ...
или e_google_pay_canceled
```

### Сервер (Laravel)

```
GOOGLE_PAY_CHARGE { "orderReference": "...", "response": "... transactionStatus ..." }
```

При успехе в ответе WFP: `WaitingAuthComplete` или `Approved`.  
При ошибке настройки merchant GPay возможен `reasonCode: 1129` (как на эмуляторе с TEST-токеном).

## 6. Эмулятор (справочно)

- GPay всегда **TEST** — шторка с Test Card и предупреждением о тесте **нормальна**.
- Холд на WayForPay с TEST-токеном часто **не проходит** (1129) — **не считается провалом release-приёмки**.
- На эмуляторе можно проверять: выбор GPay, открытие шторки, отмену, отсутствие заказа при ошибке холда.
